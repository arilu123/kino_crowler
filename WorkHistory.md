### Решение 1. Способ доступа к сайту
- Прямой автоматический запрос (`WebFetch`) к `kinopoisk.ru/film/...` даёт **302 → `sso.passport.yandex.ru`** (анти-бот/авторизация Яндекса). Без залогиненного браузера страницу не получить.
- **Вывод:** данные снимаем из живого браузера пользователя (он залогинен, капча пройдена).
- Конкретный способ массового сбора пока **не зафиксирован** (выбран режим «сначала эксперимент»). Кандидаты на будущее:
  1. Console/userscript JS на открытой странице (сейчас — основной для эксперимента).
  2. Browser extension, собирающая данные по мере просмотра.
  3. Прямой запрос к GraphQL API `https://graphql.kinopoisk.ru/graphql/` — сайт сам так грузит данные. Требует cookies/headers сессии. **Самый эффективный путь для масштаба — исследовать отдельно.**

### Решение 2. Где лежат данные на странице
- `__NEXT_DATA__` (Next.js) присутствует, но `apolloState.data` **пустой** — данные подгружаются клиентом через GraphQL после гидрации, в статический HTML не попадают.
- Поэтому работаем с **живым DOM** (пользователь сохраняет через "Edit as HTML" / `document.documentElement.outerHTML`).
- В DOM есть два надёжных источника:
  - **ld+json** (`<script type="application/ld+json">`, schema.org `Movie`) — чистая структура: name, alternateName, datePublished, genre, countryOfOrigin, timeRequired, contentRating, aggregateRating, description, image, director[], actor[] (топ-10), producer[].
  - **Таблица «О фильме»** `data-test-id="encyclopedic-table"` — строки со стабильными `data-test-id` (year, countries, genres, tagline, directors, writers, producers, operators, composers, designers, filmEditors, budget, usaBox, worldBox, rusBox, audience, ruPremiere, worldPremieres, dvdRelease, ageRestriction, ratingMPAA, duration).
- **Якоря:** классы захешированы (`styles_xxx`), опираемся на `data-test-id` и ссылки `/name/ID/` (дают id+имя персон) и `/film/ID/`.

### Решение 7. Бейдж статуса, самообнаружение атрибутов, посещённость (2026-06-06)
1. **Оповещение**: content-script рисует плавающий бейдж (правый низ) со статусом и ID:
   `⏳ запись #id…` → `✓ #id записан` (новый, зелёный) / `↻ #id обновлён · был <дата>` (синий).
   Писатель возвращает `isNew` (через `crawled_at = updated_at`) и `firstSeen`.
2. **Самообнаружение новых атрибутов** (защита от невиданных полей на N-м фильме):
   - экстрактор снимает ВСЕ строки таблицы (`_allTable` = каждый `:scope > [data-test-id]`)
     и все ключи ld (`_ldKeys`), кладёт в `raw`;
   - писатель сверяет с `KNOWN_TABLE_KEYS`/`KNOWN_LD_KEYS`; впервые встреченный ключ →
     `discovered_attrs(source, key, first_film_id, first_value)` ON CONFLICT DO NOTHING.
   - Ревизия: `SELECT * FROM discovered_attrs ORDER BY first_seen DESC;` → промотать в колонки.
3. **Посещённость / антиповтор**:
   - источник правды — наличие в `films`;
   - эндпойнт `POST /known {ids}` → какие из id уже в БД;
   - content-script подсвечивает на странице ссылки на посещённые фильмы (зелёная рамка);
   - консольный API `window.kp`: `status()`, `links()` ({all, known, unvisited}), `go(id)`, `refresh()`.
   - Проверено: isNew, discovery (поймал spectators/newField/weirdLdKey), /known — всё ок.

### Решение 11. ВАЖНО: ld+json не обновляется на SPA → извлечение DOM-first (2026-06-06)
Симптом (по логам пользователя): при переходе по ссылке 447301→397667 скрипт виснет:
`ждём синхронизации: url id = 397667 | ld Movie на странице: [447301]` — навсегда.
**Причина:** `ld+json` рендерится сервером только при ПОЛНОЙ загрузке; при SPA-переходе сайт
обновляет видимый DOM (таблица, заголовок, актёры), а `ld+json` остаётся от старого фильма.
Значит и проверка синхронизации по `ld.url`, и все поля из ld на SPA-странице были бы от старого фильма.
**Фикс — извлечение из DOM (всё перерисовывается при переходе):**
- Синхронизация (id текущего фильма) — по ТЕЛУ: ссылка `a[href*="/film/ID/cast"]` (запасной — canonical).
  `consistentId()` = адрес совпал с `domFilmId()`. От ld больше не зависит.
- Якоря DOM: `h1[itemprop=name]` (назв.+год), `[class*=originalTitle]` (ориг.),
  `a[itemprop=actor]` (актёры, их 15 vs 10 в ld), `[data-tid="kp-movie-rating.rating-value"]` (рейтинг),
  `button[aria-label*="оцен"]` (кол-во оценок), `meta[name=description]` (описание, data-next-head),
  `meta[property=og:image]` (постер-ссылка), таблица `encyclopedic-table` (жанры/страны/сборы/crew/…).
- `ld` используется ТОЛЬКО когда совпадает с адресом (полная загрузка) — для точного рейтинга/описания
  и для `_ld` (award/video). На SPA `_ld=null` (устаревший ld не пишем).
- Проверено симуляцией по сохранённому HTML: все якоря дают верные значения.
- Диагностические логи `[kp]` оставлены (DEBUG=true), при необходимости выключить.

### Решение 13. КРИТИЧНО: убран MutationObserver — он вешал страницу (2026-06-06)
Симптом: с расширением страница фильма виснет — переход по ссылкам и перезагрузка не работают;
без расширения всё ок.
**Причина:** `MutationObserver` следил за всем деревом, а наши же правки DOM (подсветка ссылок
через opacity, перерисовка панели innerHTML) порождали новые мутации → обработчик снова → шторм,
забивающий главный поток React-страницы → навигация/reload подвисают.
**Фикс:** observer полностью удалён. Вся работа — один `setInterval(1500мс)`:
`schedule()` каждый тик; `markLinks()`/`sendDiscover()` через тик (≈3с). Обёрнуто в try/catch.
SPA-переходы ловятся опросом (в пределах ~1.5–2с). Запись DOM (подсветка) — только по новым ссылкам.

### Решение 16. Парсер полного каста /film/{id}/cast/ (2026-06-07)
Образец: cast-301.html (Матрица). Это КЛАССИЧЕСКИЙ формат (ассеты st.kp.yandex.net), не Next.js,

  upsert персон (+name_orig), пересборка film_credits (с character/ord), `full_cast_fetched=true`,

### Решение 17. Парсер дат /film/{id}/dates/ (2026-06-07)
Образец dates-301.html (классический формат, таблица). Строка даты `<tr>` (листовая):
`div.flag.flagN` (id страны), `td.news b` (дата), `td[270]`: ссылка страны `/…country…/ID/` +
`<small>`=тип (per-row), `td[4]`: `<small>`=примечание (зрители/прокатчик).
- ВАЖНО: `smalls[0]` — это per-row квалификатор («Переиздание»/«Интернет»/«Премьера на DVD»/
  фестиваль/регион), а НЕ секционный тип. Перенос вперёд (первая версия) расползался — убран.
  Пусто = театральная премьера. Подтверждено на 301: 101 запись (67 премьер, 21 переиздание, 4 интернет…).
- content.js: `extractDates()` берёт листовые `<tr>` (`!tr.querySelector('tr')`, есть div.flag и td.news b),
  парсит дату рус-месяцами → ISO (без дня → NULL). `maybeSendDates()` на `^/film/\d+/dates/?$`, дедуп SENT_DATES.
- БД: таблица `film_dates(film_id, ord, date_text, date, country_id, country, type, note)` PK(film_id,ord),
  флаг `films.dates_fetched`. Эндпойнт `/dates` (saveDates): пересборка DELETE→INSERT, ставит dates_fetched.
- Проверено вживую (3 записи, ISO-дата, тип, примечание).
