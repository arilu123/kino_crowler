### Решение 1. Способ доступа к сайту
- Прямой автоматический запрос (`WebFetch`) к `kinopoisk.ru/film/...` даёт **302 → `sso.passport.yandex.ru`** (анти-бот/авторизация Яндекса). Без залогиненного браузера страницу не получить.
- **Вывод:** данные снимаем из живого браузера пользователя (он залогинен, капча пройдена).
- Конкретный способ массового сбора пока **не зафиксирован** (выбран режим «сначала эксперимент»). Кандидаты на будущее:
  1. Console/userscript JS на открытой странице (сейчас — основной для эксперимента).
  2. Browser extension, собирающая данные по мере просмотра.
  3. Прямой запрос к GraphQL API `https://graphql.kinopoisk.ru/graphql/` — сайт сам так грузит данные. Требует cookies/headers сессии. **Самый эффективный путь для масштаба — исследовать отдельно.**

### Решение 2. Где лежат данные на странице
- `__NEXT_DATA__` (Next.js) присутствует, но `apolloState.data` **пустой** — данные подгружаются клиентом через GraphQL после гидрации, в статический HTML не попадают.
- Поэтому работаем с **живым DOM** (пользователь сохраняет через "Edit as HTML" / `document.documentElement.outerHTML`).
- В DOM есть два надёжных источника:
  - **ld+json** (`<script type="application/ld+json">`, schema.org `Movie`) — чистая структура: name, alternateName, datePublished, genre, countryOfOrigin, timeRequired, contentRating, aggregateRating, description, image, director[], actor[] (топ-10), producer[].
  - **Таблица «О фильме»** `data-test-id="encyclopedic-table"` — строки со стабильными `data-test-id` (year, countries, genres, tagline, directors, writers, producers, operators, composers, designers, filmEditors, budget, usaBox, worldBox, rusBox, audience, ruPremiere, worldPremieres, dvdRelease, ageRestriction, ratingMPAA, duration).
- **Якоря:** классы захешированы (`styles_xxx`), опираемся на `data-test-id` и ссылки `/name/ID/` (дают id+имя персон) и `/film/ID/`.

### Решение 7. Бейдж статуса, самообнаружение атрибутов, посещённость (2026-06-06)
1. **Оповещение**: content-script рисует плавающий бейдж (правый низ) со статусом и ID:
   `⏳ запись #id…` → `✓ #id записан` (новый, зелёный) / `↻ #id обновлён · был <дата>` (синий).
   Писатель возвращает `isNew` (через `crawled_at = updated_at`) и `firstSeen`.
2. **Самообнаружение новых атрибутов** (защита от невиданных полей на N-м фильме):
   - экстрактор снимает ВСЕ строки таблицы (`_allTable` = каждый `:scope > [data-test-id]`)
     и все ключи ld (`_ldKeys`), кладёт в `raw`;
   - писатель сверяет с `KNOWN_TABLE_KEYS`/`KNOWN_LD_KEYS`; впервые встреченный ключ →
     `discovered_attrs(source, key, first_film_id, first_value)` ON CONFLICT DO NOTHING.
   - Ревизия: `SELECT * FROM discovered_attrs ORDER BY first_seen DESC;` → промотать в колонки.
3. **Посещённость / антиповтор**:
   - источник правды — наличие в `films`;
   - эндпойнт `POST /known {ids}` → какие из id уже в БД;
   - content-script подсвечивает на странице ссылки на посещённые фильмы (зелёная рамка);
   - консольный API `window.kp`: `status()`, `links()` ({all, known, unvisited}), `go(id)`, `refresh()`.
   - Проверено: isNew, discovery (поймал spectators/newField/weirdLdKey), /known — всё ок.

### Решение 11. ВАЖНО: ld+json не обновляется на SPA → извлечение DOM-first (2026-06-06)
Симптом (по логам пользователя): при переходе по ссылке 447301→397667 скрипт виснет:
`ждём синхронизации: url id = 397667 | ld Movie на странице: [447301]` — навсегда.
**Причина:** `ld+json` рендерится сервером только при ПОЛНОЙ загрузке; при SPA-переходе сайт
обновляет видимый DOM (таблица, заголовок, актёры), а `ld+json` остаётся от старого фильма.
Значит и проверка синхронизации по `ld.url`, и все поля из ld на SPA-странице были бы от старого фильма.
**Фикс — извлечение из DOM (всё перерисовывается при переходе):**
- Синхронизация (id текущего фильма) — по ТЕЛУ: ссылка `a[href*="/film/ID/cast"]` (запасной — canonical).
  `consistentId()` = адрес совпал с `domFilmId()`. От ld больше не зависит.
- Якоря DOM: `h1[itemprop=name]` (назв.+год), `[class*=originalTitle]` (ориг.),
  `a[itemprop=actor]` (актёры, их 15 vs 10 в ld), `[data-tid="kp-movie-rating.rating-value"]` (рейтинг),
  `button[aria-label*="оцен"]` (кол-во оценок), `meta[name=description]` (описание, data-next-head),
  `meta[property=og:image]` (постер-ссылка), таблица `encyclopedic-table` (жанры/страны/сборы/crew/…).
- `ld` используется ТОЛЬКО когда совпадает с адресом (полная загрузка) — для точного рейтинга/описания
  и для `_ld` (award/video). На SPA `_ld=null` (устаревший ld не пишем).
- Проверено симуляцией по сохранённому HTML: все якоря дают верные значения.
- Диагностические логи `[kp]` оставлены (DEBUG=true), при необходимости выключить.

### Решение 13. КРИТИЧНО: убран MutationObserver — он вешал страницу (2026-06-06)
Симптом: с расширением страница фильма виснет — переход по ссылкам и перезагрузка не работают;
без расширения всё ок.
**Причина:** `MutationObserver` следил за всем деревом, а наши же правки DOM (подсветка ссылок
через opacity, перерисовка панели innerHTML) порождали новые мутации → обработчик снова → шторм,
забивающий главный поток React-страницы → навигация/reload подвисают.
**Фикс:** observer полностью удалён. Вся работа — один `setInterval(1500мс)`:
`schedule()` каждый тик; `markLinks()`/`sendDiscover()` через тик (≈3с). Обёрнуто в try/catch.
SPA-переходы ловятся опросом (в пределах ~1.5–2с). Запись DOM (подсветка) — только по новым ссылкам.

### Решение 16. Парсер полного каста /film/{id}/cast/ (2026-06-07)
Образец: cast-301.html (Матрица). Это КЛАССИЧЕСКИЙ формат (ассеты st.kp.yandex.net), не Next.js,

  upsert персон (+name_orig), пересборка film_credits (с character/ord), `full_cast_fetched=true`,

- **Проверено end-to-end (на реальном 37859):** все колонки заполнены (male, 1974-11-11, Лос-Анджелес/

### Решение 27. Архив исходного HTML: page_html (2026-06-08)
Пользователь: складывать исходный HTML всех разбираемых страниц в отдельную таблицу; ПЕРЕД парсингом
страницу обязательно сохранять/обновлять в исходники. Раньше HTML в БД не попадал — был только
структурный `raw` (films/persons), а у подстраниц снимка не было вовсе.
- БД: `page_html(url text PK, raw_html text, fetched_at)`. url = `origin+path` (без query, чтобы
  вкладки/якоря не плодили дубли; разные пути вроде `/box/rus/` — это РАЗНЫЕ url, и хорошо).
  Отдельный gzip НЕ нужен: Postgres хранит большие строки через TOAST со сжатием
  (проверено: 535KB→299KB, 1.2MB→444KB, 1.4MB→558KB, ~2–3×).
- content.js: `maybeSaveHtml()` — ПЕРВЫЙ вызов в `tick()` (до парсеров). Срабатывает только на
  разбираемых страницах (RE_FILM_MAIN | RE_FILM_SUB cast|dates|box(/country)|studio|other|keywords|
  awards | RE_NAME_MAIN). Готовность: React-страницы (главная фильма, персона) — тело должно
  соответствовать адресу (`consistentId()` / `domNameId()`), классические подстраницы — всегда готовы.
  Дедуп per-url за сессию (`SENT_HTML`); при ошибке флаг снимается → повтор на следующем тике.
  Снимок = `document.documentElement.outerHTML` (живой DOM, как и парсеры).
- background `html`→`/html`. writer `saveHtml()` (upsert по url, обновляет raw_html+fetched_at) +
  эндпойнт `/html`. `readBody` без лимита размера — большие HTML проходят.
- Порядок «перед парсингом»: в тике html отправляется ПЕРВЫМ; html-сейв и парс — независимые POST,
  строгая атомарность не гарантируется, но исходник сохраняется всегда (с ретраем) → принцип соблюдён.
- Проверено end-to-end: POST awards-301/name-37859/cast-301 → 3 строки, повторный POST awards не
  плодит дубль (upsert), размеры/сжатие сходятся.
- ⚠️ Объём: при тысячах страниц таблица будет крупной (даже со сжатием ~0.3–0.6 МБ/страница).
  Это осознанная страховка по требованию пользователя.

### Решение 28. Оптимистичный коммит: исходник + данные параллельно, ✓ только если оба (2026-06-08)
Идея пользователя: не ждать записи raw_html перед парсингом — пускать её АСИНХРОННО параллельно с
парсингом, в конце дождаться обоих и показывать «✓» только при полном успехе. Принял с уточнением:
страницу помечаем обработанной (в `SENT_*`/`SEEN`) лишь при успехе ОБОИХ — иначе повтор на след. тике
(безопасно: все записи идемпотентны — upsert / DELETE+INSERT, html — upsert по url).
- Прежняя схема (реш. 27): отдельный `maybeSaveHtml()` первым в тике, fire-and-forget, бейдж от парсера
  не зависел от исходника. Заменено.
- content.js: `sendMsg(msg)` — промисификация `chrome.runtime.sendMessage`. `ensureHtmlSaved()` —
  мемоизированный per-url Promise<bool> (успех кэшируется, ошибка нет → повтор); снимает
  `document.documentElement.outerHTML` текущей (уже синхронной) страницы. `commitPage(sentSet, key,
  busy, ok, fail, msg, log)` — общий помощник: `setBadge(busy)` → старт `ensureHtmlSaved()` (не ждём) →
  `await sendMsg(msg)` (парс параллельно) → `await htmlP` → ✓ только если оба, иначе ✗ и `sentSet.delete`.
- Все 9 точек отправки переведены на эту модель: главная (`send()` — async, с сохранением логики
  isNew/↻/newAttrs/refreshQueue) + 7 подстраниц (cast/dates/box/studio/other/keywords/awards) +
  персона. Дублирующийся callback-код убран (−~90 строк), поведение унифицировано.
- `maybeSaveHtml`/`SENT_HTML`/`htmlPageReady`/`RE_*` удалены; вызов из тика убран. Эндпойнты writer и
  маршруты background не менялись.
- Проверено: `node --check` всех трёх файлов; конкурентный POST (`/html` + `/awards` одновременно в
  потоках) — оба успешны (pg Pool тянет параллельные запросы). Реальные награды 301 восстановлены (26).
- Браузерная часть (порядок бейджей вживую) — финальная сверка при заходе расширением, как обычно.

### Решение 29. Постоянный показ нерешённых атрибутов + чистка тестового «weird» (2026-06-08)
Баг (пользователь): атрибут `weird` висел `status='new'`, но в панели не показывался → когда-то
мелькнул на первой встрече и был пропущен. Причина: плашка `🆕` рисовалась ТОЛЬКО из `newAttrs`
(ключи, впервые вставленные в этом конкретном сохранении) — разовое событие. После первой встречи
ключ «известен», `newAttrs` пуст, плашка больше не появлялась, хотя триажа не было.
**Фикс — рисуем из БД, постоянно:**
- writer: `/queue` (`queueFilms`) дополнительно отдаёт `pendingAttrs` = `SELECT … FROM discovered_attrs
  WHERE status='new'` (source, key, first_film_id, first_value).
- content.js: `setNewAttrs(list)` → `setPendingAttrs(list)` — рисует ВСЕ нерешённые (source:key, пример
  значения, ссылка на #first_film_id), оранжевый блок виден, пока список непуст. Вызывается из
  `renderQueue` (т.е. при каждом `refreshQueue`). Из `send()` разовые `setNewAttrs([])`/`setNewAttrs(
  r.newAttrs)` убраны (остался только `console.warn` о первой встрече). В тик добавлен
  `refreshQueue()` каждые 8 тиков (~12 c) — раньше очередь обновлялась лишь после save/discover,
  а `sendDiscover` рано выходит, если новых ссылок нет → панель могла не обновляться. Теперь надёжно.
- **Откуда `weird`:** тестовый мусор от 2026-06-06 (реш. 7, проверка самообнаружения «поймал
  spectators/newField/weirdLdKey»). `first_film_id=999777` — выкидной тестовый id; фильма нет в `films`,
  ни в одном `raw->'_ld'` ключа `weird` нет. Это НЕ атрибут Сайта. Удалён

- Проверено: `/queue` до чистки отдавал weird в pendingAttrs, после — пусто; осталось promoted 5,
  deferred 2, new 0. node --check ok.

### Решение 30. Разбиение монолитов content.js / writer.js на модули (2026-06-08)
- **Проблема:** `content.js` (1160 строк) и `writer.js` (803) разрослись — дорого читать целиком
  (токены) и тяжело ориентироваться. Логика уже была чётко разбита секциями-комментариями.
- **content-script (extension/):** убрана обёрточная IIFE, код разрезан на 7 файлов. Content-скрипты
  одного `content_scripts.js[]` исполняются в ОДНОМ isolated-world и делят глобальную лексическую
  область — top-level `const`/`let`/`function` видны между файлами. Все межфайловые вызовы — в рантайме
  (внутри `tick()`/обработчиков), поэтому порядок важен только для автозапуска: `content-main.js`
  (цикл `tick`) грузится ПОСЛЕДНИМ. Модули: `content-core.js` (состояние, утилиты, id/ld, парс дат,
  `commitPage`/`sendMsg`/`ensureHtmlSaved`), `content-panel.js`, `content-film.js` (`extract`/`send`/
  `schedule`), `content-discover.js`, `content-subpages.js` (cast/dates/box/studio/other/keywords/awards),
  `content-person.js`, `content-main.js` (`window.kp` + `tick`). `RU_MONTHS`/`parseRuDate` перенесены в
  core (нужны и dates, и person). Порядок прописан в `manifest.json`.
- **writer (server/):** CommonJS → `require`/`module.exports`. `db.js` (пул `pool`), `saves.js` (все
  `save*` + самообнаружение атрибутов), `queue.js` (`knownFilmIds`/`discoverLinks`/`queueFilms`),
  `writer.js` остался точкой входа (`main` в package.json): http-сервер + роутинг.
- **Проверка:** `node --check` всех файлов ✓; сверка символов content (рабочая копия vs сумма модулей) —
  потерь/задвоений нет; тестовый writer на :8799 — `/queue`, `/known` (вернул `[78871]`), `/movie` с битым
  телом (`{"error":"no movie id"}`) отработали против реальной БД. Доки обновлены (README, movie-info-list).
- ⚠️ Действие юзера: нажать «обновить» у расширения в `chrome://extensions` (изменился список файлов).
  Запущенный старый `writer.js` (pid был 7767 на :8787) перезапустить, чтобы подхватить модули.
